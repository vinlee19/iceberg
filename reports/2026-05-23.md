## 二、v4 相对路径特性背景

在理解今日两个 PR 之前，有必要了解 v4 相对路径特性的动机：

```
Iceberg 元数据文件中存储了大量文件路径（数据文件、删除文件、manifest等）
传统 v3 及以前版本中，所有路径均为绝对路径（如 s3://bucket/db/table/...）

问题：
  ┌─────────────────────────────────────────────────────┐
  │  当表需要迁移/重命名时，必须重写所有元数据文件       │
  │  因为绝对路径包含了旧的 bucket/prefix 信息          │
  └─────────────────────────────────────────────────────┘

v4 解决方案：支持相对路径
  ┌─────────────────────────────────────────────────────┐
  │  路径可以相对于表的 location 存储                   │
  │  如: data/00000-0.parquet  (相对)                   │
  │  迁移时只需更新 table location，元数据无需重写      │
  └─────────────────────────────────────────────────────┘
```

#### 3.2 路径解析规则

```
路径分类判断（决策流程）：
                        ┌──────────────────┐
                        │   读取路径字符串  │
                        └────────┬─────────┘
                                 │
                    ┌────────────▼────────────┐
                    │  是否包含 URI scheme?   │
                    │  (s3:, hdfs:, gs: 等)  │
                    └────────┬────────┬───────┘
                             │ 是     │ 否
                     ┌───────▼──┐  ┌──▼────────────────┐
                     │ 绝对路径  │  │    相对路径         │
                     │ 直接使用  │  │ 与 table location  │
                     └──────────┘  │ 拼接后使用          │
                                   └────────────────────┘
```

#### 3.3 路径解析示例表（规范中新增）

| 类型 | 格式版本 | Table Location | File Path | 解析结果 |
|:-----|:---------|:---------------|:----------|:---------|
| 相对路径 | v4 | `s3://bucket/db/table` | `data/00000-0.parquet` | `s3://bucket/db/table/data/00000-0.parquet` |
| 绝对路径 | v4 | `s3://bucket/db/table` | `hdfs://wh/db/table/data/file` | `hdfs://wh/db/table/data/file`（保持不变）|
| 分隔符重复（警告）| v4 | `s3://bucket/db/table/` | `data/file.parquet` | `s3://bucket/db/table//data/file.parquet` ⚠️ |
| 历史全路径 | v3 及更早 | `s3://bucket/db/table` | `s3://bucket/db/table/data/file` | `s3://bucket/db/table/data/file` |

> ⚠️ **注意**: 规范明确要求 `table location` 末尾不应以 `/` 结尾，`file path` 开头不应以 `/` 开头，否则会产生双斜杠 `//`。

#### 3.4 v4 表元数据结构变化

v4 中 `location` 字段从 **必填** 变为 **可选**：

```
v3 及以前:
  table metadata {
    "location": "s3://bucket/db/table",  ← 必填
    ...
  }

v4:
  table metadata {
    "location": "s3://bucket/db/table",  ← 可选（当不存在时，由 catalog 提供）
    ...
  }
```

这一改变意味着 **catalog** 可以管理和控制表的位置，增强了灵活性。

#### 3.5 向后兼容性规则

**读取 v3 数据（为 v4 使用）：**
- 所有路径字段都是全限定路径，按绝对路径解析
- 没有 URI scheme 前缀的路径需要补充 scheme

**写入 v4 数据：**
- location 字段现在可选，存在时必须是绝对路径
- 所有元数据位置字段可包含相对路径
- 推荐对表 location 下的文件使用相对路径

#### 4.3 `resolveLocation` 方法实现解析

```java
public static String resolveLocation(String tableLocation, String location) {
    if (hasScheme(location)) {
        return location;          // 绝对路径，直接返回
    }
    return tableLocation + PATH_SEPARATOR + location;  // 拼接相对路径
}
```

```
调用示例：
  tableLocation = "s3://bucket/db/table"
  location      = "data/00000-0.parquet"
  
  hasScheme("data/00000-0.parquet") → false （无 scheme）
  
  返回: "s3://bucket/db/table" + "/" + "data/00000-0.parquet"
       = "s3://bucket/db/table/data/00000-0.parquet"  ✓
```

#### 4.4 `relativizeLocation` 方法实现解析

```java
public static String relativizeLocation(String tableLocation, String location) {
    int prefixLength = tableLocation.length();
    if (location.length() > prefixLength
        && location.startsWith(PATH_SEPARATOR, prefixLength)  // 确认紧跟 "/"
        && location.startsWith(tableLocation)) {              // 确认前缀匹配
        return location.substring(prefixLength + PATH_SEPARATOR.length());
    }
    return location;  // 不匹配则原样返回
}
```

```
调用示例（成功相对化）：
  tableLocation = "s3://bucket/db/table"
  location      = "s3://bucket/db/table/data/00000-0.parquet"
  
  prefixLength = 22
  location[22] == '/'  → true
  location.startsWith("s3://bucket/db/table") → true
  
  返回: "data/00000-0.parquet"  ✓

调用示例（不相对化的情况 - 共享前缀但不是子路径）：
  tableLocation = "s3://bucket/db/table"
  location      = "s3://bucket/db/table_v2/data/file.parquet"
  
  location[22] == '_'  → false（不是 '/'）
  
  返回: "s3://bucket/db/table_v2/data/file.parquet"（原样返回）✓
```

> **关键设计**: `location.startsWith(PATH_SEPARATOR, prefixLength)` 这个检查防止了"共享前缀但非子目录"的误判（如 `table` 与 `table_v2`）。

#### 4.5 URI Scheme 检测实现（遵循 RFC 3986）

```java
// RFC 3986 Section 3.1 定义:
// scheme = ALPHA *( ALPHA / DIGIT / "+" / "-" / "." )
// 即: 首字符必须是字母, 后续可以是字母/数字/+/-/.

private static boolean hasScheme(String location) {
    for (int i = 0; i < location.length(); i += 1) {
        char ch = location.charAt(i);
        if (ch == ':') {
            return i > 0;  // 找到冒号，且冒号前有字符 → 有 scheme
        }
        if (!isSchemeChar(ch, i)) {
            return false;  // 遇到非 scheme 字符 → 无 scheme
        }
    }
    return false;
}
```

```
合法 scheme 示例:
  "s3://..."      → scheme = "s3"      ✓
  "hdfs://..."    → scheme = "hdfs"    ✓
  "gs://..."      → scheme = "gs"      ✓
  "file://..."    → scheme = "file"    ✓
  "s3+ext://..."  → scheme = "s3+ext"  ✓  (含 +)

特殊处理 - 含冒号的相对路径（常见于分区值）：
  "data/partition=key:value/file.parquet"
    → 遍历到 '=' 时已不满足 isSchemeChar → hasScheme = false ✓
  "metadata/snap-123:456.avro"
    → 遍历到 '-' 时（position=4）已不满足 isSchemeChar → hasScheme = false ✓
```

这个实现巧妙地处理了 **分区路径中含有冒号** 的边界情况（如 `partition=key:value`），不会被误识别为绝对路径。

| 场景 | 影响 |
|:-----|:-----|
| **只读 v3 表** | 无影响，v3 路径全为绝对路径 |

- `TableMetadataParser` — 元数据序列化/反序列化
- `ManifestWriter` — manifest 文件写入
- `ManifestReader` — manifest 文件读取
- `BaseTransaction` — 事务提交路径处理
