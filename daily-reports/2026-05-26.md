### PR #16400 — REST Spec: Add unregister table endpoint

#### 背景与动机

Apache Iceberg REST Catalog 已有 `registerTable` 端点，允许将一张表从外部（如 Hive、其他 Catalog）注册到 REST Catalog 中。这是 Hive 迁移到 REST 的核心操作。

然而，当用户需要**从一个 REST Catalog 迁移到另一个 REST Catalog** 时，缺乏对应的"取消注册"接口。如果直接调用 `DELETE /tables/{table}`，表的数据和元数据文件将被物理删除，无法安全迁移。

本 PR 填补了这一空白，实现了 `registerTable` 的反向操作：**从 Catalog 注销一张表，但保留所有数据和元数据文件**。

#### 变更详情

**1. 新增 REST API 端点**

```
POST /v1/{prefix}/namespaces/{namespace}/tables/{table}/unregister
```

- 位于 `open-api/rest-catalog-open-api.yaml`（+50 行）
- 使用 **空 POST 请求体**，参数通过 URL 路径传递（`prefix`、`namespace`、`table`）
- 支持 `idempotency-key` 请求头，保证幂等性
- 端点设计遵循 REST 资源语义：挂载在 `/{table}/unregister` 子路径下，而非 `register` 旁边

**2. 新增响应模型**

```yaml
# UnregisterTableResult schema
UnregisterTableResult:
  required:
    - metadata-location
    - metadata
  properties:
    metadata-location:
      type: string        # 表注销时最后一次 metadata 的位置
    metadata:
      $ref: '#/components/schemas/TableMetadata'  # 完整的表元数据
```

成功响应（HTTP 200）返回：
- `metadata-location`：注销时刻表的最后一次 metadata 文件路径
- `metadata`：完整的 TableMetadata 对象（包含所有历史提交）

> **关键语义保证：** 注销成功后，该 Catalog 中所有后续对该表的写入提交**必须失败**。这确保了迁移窗口期内数据的一致性。

**3. 错误处理**

| HTTP 状态码 | 说明 |
|------------|------|
| 200 | 成功注销，返回 `UnregisterTableResult` |
| 400 | 请求参数错误 |
| 401 | 未授权 |
| 403 | 禁止访问 |
| 404 | 表不存在（`NoSuchTableException`） |
| 419 | 认证超时 |
| 503 | 服务不可用 |
| 5XX | 服务器内部错误 |

**4. Python 辅助脚本**

`open-api/rest-catalog-open-api.py` 新增 13 行，用于生成/验证 OpenAPI 规范文件。

#### 架构示意

```
迁移流程：REST Catalog A → REST Catalog B
─────────────────────────────────────────────────────

     Catalog A                           Catalog B
┌──────────────────┐                ┌──────────────────┐
│  table: my_table │                │                  │
│  (已注册)        │                │  (未注册)         │
└────────┬─────────┘                └──────────────────┘
         │
         │ POST /tables/my_table/unregister
         ▼
┌──────────────────┐
│  返回:            │
│  metadata-location│  ──────────────►  POST /register
│  metadata         │                   { metadata-location }
└──────────────────┘                ┌──────────────────┐
                                    │  table: my_table │
     Catalog A                      │  (已注册)        │
  后续写入全部拒绝 ✗                 └──────────────────┘
                                         Catalog B
```

#### 影响范围

- **REST Catalog 实现方（Server Side）**：需要实现 `unregisterTable` 操作的业务逻辑（停止追踪表、拒绝后续写入）
- **REST Catalog 客户端**：可以通过 API 安全地将表迁移到新 Catalog，无需担心数据丢失
- **Iceberg 版本兼容**：该 PR 仅修改 OpenAPI 规范，Java 核心实现待后续 PR 跟进

#### 总结

这是 Iceberg REST Catalog 生态中关于**跨 Catalog 表迁移**的重要基础设施补全，与 `registerTable` 形成完整的迁入/迁出对称接口，有助于提升 Iceberg 在多 Catalog 混合部署场景下的数据管理灵活性。

### Issue #16567 — Variant 列分片（Shredding）写入时 Metrics 收集崩溃

| 属性 | 详情 |
|------|------|
| **提交者** | [@soumilshah1995](https://github.com/soumilshah1995) |
| **创建时间** | 2026-05-26 15:19:48 UTC |
| **标签** | `bug` 🔴 |
| **状态** | Open |
| **Issue 链接** | https://github.com/apache/iceberg/issues/16567 |

#### 问题描述

在启用 **Variant Shredding**（变体类型分片写入）时，向 Iceberg v3 表 Append 数据时会在 Parquet 文件关闭阶段抛出 `java.util.NoSuchElementException`。

**复现环境：**
- Iceberg 版本：`1.11.0`（`iceberg-spark-runtime-4.0_2.13`）
- Spark 版本：Spark 4.0.2-amzn-0（AWS EMR PySpark）
- Catalog：Iceberg REST Catalog → Amazon S3 Tables（`SparkCatalog`, `S3FileIO`）
- 配置：`spark.sql.iceberg.shred-variants=true`

#### 报错堆栈

```
java.util.NoSuchElementException
    at com.google.common.collect.Iterables.getOnlyElement(Iterables.java:262)
    at ParquetMetrics$MetricsVisitor$MetricsVariantVisitor.value(ParquetMetrics.java:480)
    at ParquetMetrics$MetricsVisitor$MetricsVariantVisitor.value(ParquetMetrics.java:410)
    at ParquetVariantVisitor.visitValue(ParquetVariantVisitor.java:215)
    at ParquetVariantVisitor.visitObjectFields(ParquetVariantVisitor.java:265)
    at ParquetVariantVisitor.visitValue(ParquetVariantVisitor.java:226)
    at ParquetVariantVisitor.visit(ParquetVariantVisitor.java:183)
    at ParquetMetrics$MetricsVisitor.variant(ParquetMetrics.java:362)
    at TypeWithSchemaVisitor.visitVariant(TypeWithSchemaVisitor.java:221)
    at ParquetMetrics.metrics(ParquetMetrics.java:101)
    at ParquetWriter.metrics(ParquetWriter.java:173)
    at DataWriter.close(DataWriter.java:90)
    at RollingFileWriter.closeCurrentWriter(RollingFileWriter.java:126)
    at SparkWrite$PartitionedDataWriter.commit(SparkWrite.java:838)
```

#### 根因分析

`MetricsVariantVisitor.value()` 在访问 Variant 对象字段时调用了 `Iterables.getOnlyElement()`，该方法要求集合中**恰好有一个元素**。当 Variant 列启用 shredding 后，Parquet 文件结构中的子字段数量不再是 1，导致断言失败。

```
分片前（正常）：              分片后（触发 Bug）：
Variant 列                   Variant 列
└── value: [单一元素]  ✅     ├── typed_value
                             ├── untyped_value   ← getOnlyElement() 失败 ❌
                             └── metadata
```

#### 临时规避方案

在 `VARIANT` 列上禁用 metrics 收集：

```sql
ALTER TABLE catalog.test.events SET TBLPROPERTIES (
  'write.metadata.metrics.column.properties' = 'none',
  'write.metadata.metrics.column.v' = 'none'
);
```

禁用后，669M 行（216 GB）数据可正常 Append（启用 shredding）。

#### 影响范围

- **Iceberg 1.11.0** + Spark 4.0 + **VARIANT 列** + **shredding 启用** = 必现崩溃
- 大规模生产数据写入场景影响明显（该 reporter 处理 669M 行）
- 属于 Iceberg v3 新特性（VARIANT 类型 + Shredding）的重要 Bug

### PR #16570 — Spark: 修复 Scan 过滤表达式顺序不稳定导致 Exchange 无法复用

| 属性 | 详情 |
|------|------|
| **作者** | [@ahshahid](https://github.com/ahshahid) |
| **状态** | Open |
| **标签** | `spark` |
| **关联 Bug** | #16563 |
| **PR 链接** | https://github.com/apache/iceberg/pull/16570 |

**问题根因：** Spark 内部使用 `Set` 类型收集下推过滤条件，导致每次执行时过滤表达式的顺序可能不同。Iceberg `SparkScan` 在计算 `equals()` 和 `hashCode()` 时未对过滤表达式排序，导致**结构上相同的 Scan 被认为不等**，进而阻止了 Exchange（数据分发节点）的复用，影响查询性能。

**修复：** 在 `equals()` 和 `hashCode()` 中对 runtime 过滤和 data 过滤表达式进行规范化排序（基于 `Predicate.toString()` 字符串表示），确保顺序一致性。

```
修复前：
  filterSet = {filter_B, filter_A}  → hashCode_1
  filterSet = {filter_A, filter_B}  → hashCode_2  ← 同一 Scan，不同 hash ❌

修复后：
  sorted = [filter_A, filter_B]     → hashCode_X
  sorted = [filter_A, filter_B]     → hashCode_X  ← 一致 ✅
```

### PR #16571 — Spark: 支持 long 类型的 Streaming 单次微批最大行数

| 属性 | 详情 |
|------|------|
| **作者** | [@colinre](https://github.com/colinre) |
| **状态** | Open |
| **标签** | `spark` `docs` |
| **PR 链接** | https://github.com/apache/iceberg/pull/16571 |

**问题：** `streaming-max-rows-per-micro-batch` 配置项使用 `int` 类型存储，有效上限为 `Integer.MAX_VALUE`（约 21 亿行）。对于超大规模初始回填场景（如万亿行级别的表），这导致需要成千上万次微批才能完成初始化，实际无法使用流式回填。

**修复：**
- 内部使用 `long` 类型传递行数限制
- 使用 `Long.MAX_VALUE` 作为"未配置"状态的哨兵值（替换原 `Integer.MAX_VALUE`）
- 保留原 `int maxRecordsPerMicroBatch()` 方法（标记为 `@Deprecated`）
- 新增 `long maxRecordsPerMicroBatchLong()` 方法供新调用者使用
- 修复涵盖 Spark 3.5、4.0、4.1 三个版本

```
修复前：  有效行数上限 = 2,147,483,647（~21 亿）
修复后：  有效行数上限 = 9,223,372,036,854,775,807（~922 亿亿）
```
